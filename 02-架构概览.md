# 第二章：架构概览

## 2.1 整体架构设计

Claude Code 采用分层架构设计，从上到下分为五个主要层次：入口层、命令行层、核心引擎层、服务层和工具层。这种分层设计使得各层之间的职责清晰，便于理解、维护和扩展。

入口层位于整个架构的最顶端，负责处理程序启动时的各种快速路径和初始化工作。这一层的核心文件是 `src/entrypoints/cli.tsx`，它在模块加载之前就会执行一些关键的环境设置和功能标志的 Polyfill 注入。入口层的设计理念是“零开销快速路径”，对于诸如 `--version` 这类简单命令，程序可以在加载最少模块的情况下快速响应，从而提供极佳的用户体验。

命令行层由 `src/main.tsx` 主导，这是使用 Commander.js 构建的完整 CLI 应用。命令行层负责解析用户输入的各种参数和命令，进行交互模式的检测，处理深层链接协议，并注册所有的子命令。 Commander.js 的命令体系在这里被充分运用，每个命令都有对应的处理逻辑和权限控制。

核心引擎层是 Claude Code 的大脑，包含 QueryEngine 和 query.ts 两个核心文件。 QueryEngine 作为高级编排器，管理整个会话的生命周期，包括消息历史、任务状态、权限拒绝记录等。而 query.ts 则实现了真正的对话循环，负责与 API 的通信、流式响应的处理、以及工具调用的协调。这两个文件的协作构成了 Claude Code 的核心逻辑。

服务层提供了各种基础设施服务，包括 API 客户端、MCP 协议支持、分析 telemetry、状态管理等等。服务层的代码主要位于 `src/services/` 目录下，每个子目录对应一个特定的服务领域。这种模块化的服务设计使得添加新功能或修改现有功能变得相对独立，不会影响其他部分。

工具层是 Claude Code 与外部世界交互的桥梁。工具层位于 `src/tools/` 目录下，包含了五十多个工具实现，每个工具都有独立的目录。这些工具涵盖了文件操作、命令执行、网络搜索、代码编辑等方方面面。工具的设计遵循统一的接口规范，使得新增工具变得标准化和简单。

## 2.3 核心模块职责

### 2.3.1 入口点模块

**文件**: `src/entrypoints/cli.tsx`

这是真正的程序入口点，在任何其他模块加载之前执行。 cli.tsx 的主要职责包括：注入构建时常量的 Polyfill ，例如 VERSION 、 BUILD_TIME 等；设置全局环境变量如 BUILD_TARGET 和 BUILD_ENV ；实现多个零模块加载的快速路径，包括版本查询、系统提示转储等；处理特殊标志如 --claude-in-chrome-mcp 和后台会话管理。

cli.tsx 的设计核心在于“快速响应”，对于简单的命令请求，程序不需要加载完整的模块体系，只需要执行这个文件中的少量代码即可完成任务。这极大地优化了用户使用简单命令时的响应速度。

### 2.3.2 命令行模块

**文件**: `src/main.tsx`

main.tsx 是使用 Commander.js 构建的完整 CLI 应用。它负责：解析命令行参数和子命令；检测运行环境（交互式/非交互式、GitHub Actions、VSCode 等）；配置命令的 preAction 钩子，在命令执行前初始化服务；注册所有子命令（agents、auth、mcp、plugins 等）；启动 REPL 或执行非交互式任务。

main.tsx 的顶层代码会在重型模块加载之前启动性能分析和并行预读取，这是为了最大化利用模块加载的等待时间来完成一些轻量级的初始化工作。

### 2.3.3 查询引擎模块

**文件**: `src/query.ts` 和 `src/QueryEngine.ts`

这是 Claude Code 的核心逻辑所在。 QueryEngine 是高级编排器，负责管理会话级别的状态：维护消息历史、权限拒绝记录、使用统计；处理消息的序列化和反序列化，支持会话恢复；包装底层的 query() 函数，提供更高层的接口。

query.ts 实现真正的对话循环，其核心是一个 while(true) 循环：构建发送给 API 的消息列表；调用 API 获取响应；处理流式响应，实时更新 UI；检测并执行工具调用；处理各种错误和恢复逻辑（如上下文过长、输出 token 不足等）。

### 2.3.4 API 客户端模块

**文件**: `src/services/api/claude.ts`

API 客户端是 Claude Code 与 Anthropic API 通信的桥梁。它的主要功能包括：消息规范化，将内部消息格式转换为 API 格式；构建请求参数，包括系统提示、工具定义、Beta 特性标志等；处理流式响应，将原始事件转换为结构化消息；实现非流式回退，当流式响应失败时自动降级；处理缓存控制，优化 API 调用的成本和速度。

### 2.3.5 工具系统模块

**文件**: `src/Tool.ts` 和 `src/tools.ts`

Tool.ts 定义了工具的接口规范，包括工具名称、描述、输入模式、执行函数、权限检查和 UI 渲染。 tools.ts 是工具注册表，负责收集所有可用工具并提供给查询引擎使用。

工具系统的设计遵循“约定优于配置”的原则，每个工具都遵循统一的目录结构和接口定义，这使得新增工具变得非常简单和标准化。

### 2.3.6 界面渲染模块

**文件**: `src/ink.ts` 和 `src/screens/REPL.tsx`

Claude Code 使用 Ink 框架在终端中渲染 React 组件。 ink.ts 是 Ink 的封装，添加了主题支持和一些自定义 Hook。 REPL.tsx 是主交互界面，负责：显示消息历史和流式响应；处理用户输入和命令提交；显示工具权限请求；处理键盘快捷键（如 Ctrl+O 切换转录模式）。

## 2.4 数据流设计

### 整体数据流

```
用户输入
    │
    ▼
┌─────────────────┐
│   REPL.tsx     │  ◄── 用户界面，接收输入，显示输出
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ QueryEngine.ts │  ◄── 高级编排，管理会话状态
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   query.ts     │  ◄── 对话循环，处理工具调用
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌───────┐ ┌───────┐
│API    │ │Tools  │  ◄── 并行执行：API 调用和工具执行
│Client │ │System │
└───┬───┘ └───┬───┘
    │         │
    ▼         ▼
┌─────────────────┐
│ Anthropic API  │  ◄── 外部服务
└─────────────────┘
```

### 消息处理流程

用户输入首先到达 REPL 屏幕组件，组件将输入传递给 QueryEngine。 QueryEngine 调用 query.ts 中的查询函数，该函数会构建完整的��息列表（包括系统提示、历史消息和用户输入），然后调用 API 客户端发送请求。 API 客户端使用流式方式与 Anthropic API 通信，实时将响应消息yield回 query.ts。 query.ts 检测到工具调用时，会将工具调用信息传递给工具系统执行，同时继续处理后续的 API 响应。工具执行完成后，结果会被添加到消息列表中，用于下一次 API 调用。整个过程在一个循环中不断重复，直到用户主动结束对话。

### 状态更新流程

应用状态使用类似 Zustand 的 Store 管理。当某个组件需要更新状态时，它调用 setAppState 函数，传入一个更新函数。 Store 内部使用 Object.is 比较新旧状态，只有状态真正变化时才会通知订阅者。组件通过 useAppState Hook 订阅状态变化，状态更新时只有使用对应状态片段的组件会重新渲染。这种细粒度的订阅机制保证了应用的性能。

## 2.5 技术特点

### 2.5.1 内存效率优化

Claude Code 使用多种技术优化内存使用。消息系统使用虚拟滚动，只渲染当前可见的消息行，即使会话包含数千条消息也能保持流畅。状态管理采用细粒度订阅，组件只订阅自己需要的状态片段，避免不必要的渲染。工具结果有大小限制，防止单个工具输出耗尽内存。

### 2.5.2 网络效率优化

API 客户端实现了多种网络优化。预连接机制在用户开始对话前就建立与 Anthropic API 的 TCP 和 TLS 连接，减少首次响应的延迟。缓存控制使用 prompt caching 减少重复的系统提示传递。流式响应让用户能尽快看到部分结果，而不需要等待完整响应。非流式回退机制确保即使流式响应失败也能完成任务。

### 2.5.3 错误恢复机制

程序实现了多层次的错误恢复。超时保护：流式响应有超时检测，超过 30 秒无新内容会警告，超过 90 秒会中止并尝试非流式回退。上下文压缩：��提示过长时，会自动压缩历史消息，保留关键信息。输出恢复：当输出 token 不足时，会自动调整输出限制并重试。这些机制确保了对话的稳定性。

## 2.6 扩展机制

### 2.6.1 工具扩展

添加新工具只需要在 `src/tools/` 目录下创建新目录，实现以下文件：index.ts 导出工具实例，通常使用 buildTool 辅助函数创建；ToolName.ts 实现工具的核心逻辑，包括 execute 函数、inputSchema 验证和权限检查；可能还需要 UI.tsx 自定义工具在界面中的显示方式。

### 2.6.2 服务扩展

添加新服务需要在 `src/services/` 下创建新目录，实现服务的主要逻辑，然后在需要的地方导入使用。服务通常通过 AppState 的上下文提供，或者通过模块导出的方式供其他模块调用。

### 2.6.3 组件扩展

添加新界面组件需要遵循 React 和 Ink 的规范。组件应该使用 TypeScript 编写，正确处理 props 类型。如果组件需要访问应用状态，应该使用 useAppState Hook 进行订阅。对于需要响应键盘输入的组件，可以使用 useInput Hook。

## 2.7 构建系统

### 构建配置

项目使用 Bun 作为构建工具。构建配置文件包括：package.json 定义了构建命令和依赖；build.ts 是构建脚本入口；bunfig.toml 是 Bun 的配置文件；tsconfig.json 是 TypeScript 配置。

### 构建产物

运行 `bun run build` 后，会在 dist/ 目录下生成：cli.js 主入口文件，约 25MB；约 450 个 chunk 文件，用于代码分割；类型定义文件。

### 开发模式

开发模式使用 `bun run dev` 命令，它直接运行 src/entrypoints/cli.tsx，跳过构建步骤。这种方式便于快速开发和调试，但性能略低于生产构建。
