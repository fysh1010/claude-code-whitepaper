# 第十三章：限流机制与封号避免

## 13.1 限流系统概述

Claude Code 的限流系统是一个复杂但设计精良的机制，旨在帮助用户合理使用 API 配额，同时避免因过度使用而导致的各种限制和潜在风险。理解限流机制不仅可以帮助用户更好地规划使用方式，还能在遇到限流时知道如何正确处理。

Claude Code 的限流系统主要包含以下几个核心组件：首先是 API 重试机制（withRetry.ts），负责在遇到临时性错误时自动重试；其次是配额状态管理（claudeAiLimits.ts），实时追踪用户的配额使用情况；再次是限流消息处理（rateLimitMessages.ts），为用户提供清晰的限流提示；最后是 Fast Mode 降级机制，在 API 繁忙时自动切换到更快的模型。

这些组件协同工作，形成了一个完整的限流处理体系，既能保护用户避免因不了解配额情况而产生额外费用，又能在 API 负载高时提供优雅的降级体验。

## 13.2 API 重试机制详解

### 13.2.1 重试策略设计

**文件**: `src/services/api/withRetry.ts`

Claude Code 实现了一个智能的重试机制，能够处理各种临时性错误，包括 429（速率限制）、529（服务过载）等。

```typescript
// 默认最大重试次数
const DEFAULT_MAX_RETRIES = 10
const MAX_529_RETRIES = 3

// 基础延迟配置
export const BASE_DELAY_MS = 500
```

核心重试逻辑采用指数退避算法，同时添加随机抖动来避免惊群效应：

```typescript
export function getRetryDelay(
  attempt: number,
  retryAfterHeader?: string | null,
  maxDelayMs = 32000,
): number {
  // 优先使用服务器返回的 retry-after 头
  if (retryAfterHeader) {
    const seconds = parseInt(retryAfterHeader, 10)
    if (!isNaN(seconds)) {
      return seconds * 1000
    }
  }

  // 指数退避 + 随机抖动
  const baseDelay = Math.min(
    BASE_DELAY_MS * Math.pow(2, attempt - 1),
    maxDelayMs,
  )
  const jitter = Math.random() * 0.25 * baseDelay
  return baseDelay + jitter
}
```

这种设计确保了：服务器优先，优先遵循服务器的 retry-after 指示；指数退避，随着重试次数增加延迟增长；随机抖动，避免多个客户端同时重试造成拥堵；最大限制，延迟不会超过设定的最大值。

### 13.2.2 错误类型判断

系统能够识别多种可重试和不可重试的错误：

```typescript
function isTransientCapacityError(error: unknown): boolean {
  return (
    is529Error(error) || (error instanceof APIError && error.status === 429)
  )
}

// 检测 529 错误（服务过载）
export function is529Error(error: unknown): boolean {
  if (!(error instanceof APIError)) {
    return false
  }

  // 检查 529 状态码或消息中的过载错误
  return (
    error.status === 529 ||
    (error.message?.includes('"type":"overloaded_error"') ?? false)
  )
}
```

可重试的错误包括：529 服务过载、429 速率限制、500-599 服务器错误、408 请求超时、409 锁冲突、401/403 认证错误（可恢复）。

不可重试的错误包括：400 客户端错��（除特定情况外）、422 验证错误、文件未找到等客户端错误。

### 13.2.3 持久重试模式

对于无人值守的会话，系统支持特殊的持久重试模式：

```typescript
const PERSISTENT_MAX_BACKOFF_MS = 5 * 60 * 1000  // 5分钟
const PERSISTENT_RESET_CAP_MS = 6 * 60 * 60 * 1000  // 6小时
const HEARTBEAT_INTERVAL_MS = 30_000  // 30秒心跳

function isPersistentRetryEnabled(): boolean {
  return feature('UNATTENDED_RETRY')
    ? isEnvTruthy(process.env.CLAUDE_CODE_UNATTENDED_RETRY)
    : false
}
```

持久重试模式的特点包括：等待重置，如果服务器提供了重置时间，会等待到重置而不是轮询；心跳通知，长时间等待时会定期输出状态，避免会话被认为空闲；最大等待，设置了 6 小时的最大等待上限。

## 13.3 配额状态管理

### 13.3.1 配额类型

**文件**: `src/services/claudeAiLimits.ts`

Claude Code 支持多种配额类型：

```typescript
type RateLimitType =
  | 'five_hour'      // 5小时会话限制
  | 'seven_day'      // 7天周期限制
  | 'seven_day_opus' // Opus 模型周限制
  | 'seven_day_sonnet' // Sonnet 模型周限制
  | 'overage'        // 额外使用量
```

每种配额类型对应不同的使用场景：five_hour 限制单次会话的 API 调用量；seven_day 是标准的周配额；seven_day_opus 和 seven_day_sonnet 是针对特定模型的配额；overage 是超出配额后的额外计费。

### 13.3.2 配额状态检测

系统通过 API 响应头实时检测配额状态：

```typescript
function extractRawUtilization(headers: globalThis.Headers): RawUtilization {
  const result: RawUtilization = {}
  for (const [key, abbrev] of [
    ['five_hour', '5h'],
    ['seven_day', '7d'],
  ] as const) {
    const util = headers.get(
      `anthropic-ratelimit-unified-${abbrev}-utilization`,
    )
    const reset = headers.get(`anthropic-ratelimit-unified-${abbrev}-reset`)
    if (util !== null && reset !== null) {
      result[key] = { utilization: Number(util), resets_at: Number(reset) }
    }
  }
  return result
}
```

关键的响应头包括：anthropic-ratelimit-unified-status 表示配额状态（allowed/allowed_warning/rejected）；anthropic-ratelimit-unified-reset 表示配额重置时间；anthropic-ratelimit-unified-{5h/7d}-utilization 表示当前使用率；anthropic-ratelimit-unified-overage-status 表示额外使用量状态。

### 13.3.3 早期警告机制

系统实现了双重早期警告机制：

```typescript
// 早期警告配置
const EARLY_WARNING_CONFIGS: EarlyWarningConfig[] = [
  {
    rateLimitType: 'five_hour',
    windowSeconds: 5 * 60 * 60,
    thresholds: [{ utilization: 0.9, timePct: 0.72 }],
  },
  {
    rateLimitType: 'seven_day',
    windowSeconds: 7 * 24 * 60 * 60,
    thresholds: [
      { utilization: 0.75, timePct: 0.6 },
      { utilization: 0.5, timePct: 0.35 },
      { utilization: 0.25, timePct: 0.15 },
    ],
  },
]
```

早期警告的工作原理：基于时间，如果用户使用配额的速度超过时间进度，会触发警告；基于阈值，当使用率超过特定百分比时触发；双重检测，既检查服务器返回的头部，也进行本地计算作为后备。

## 13.4 限流消息处理

### 13.4.1 消息类型

**文件**: `src/services/rateLimitMessages.ts`

限流消息分为两种严重级别：

```typescript
export type RateLimitMessage = {
  message: string
  severity: 'error' | 'warning'
}
```

错误消息在以下情况显示：配额已用尽（rejected 状态）；超出额外使用量限制。警告消息在以下情况显示：使用率超过 70%（通过阈值检测）；即将达到配额限制。

### 13.4.2 消息文本示例

```typescript
// 配额用尽消息
export const RATE_LIMIT_ERROR_PREFIXES = [
  "You've hit your",
  "You've used",
  "You're now using extra usage",
  "You're close to",
  "You're out of extra usage",
] as const
```

典型的限流消息格式：

```
// 会话限制用尽
You've hit your session limit · resets in 2 hours

// 周限制用尽
You've hit your weekly limit · resets in 5 days

// 接近限制（警告）
You've used 75% of your weekly limit · resets in 3 days

// 使用额外使用量
You're now using extra usage · Your session limit resets in 30 minutes
```

### 13.4.3 消息显示位置

限流消息会在不同位置显示：错误消息在对话中显示为 assistant 消息；警告消息在状态栏显示；超出使用量通知在切换到额外使用量时显示。

## 13.5 Fast Mode 降级机制

### 13.5.1 Fast Mode 概述

Fast Mode 是 Claude Code 的一个优化功能，使用更小的模型来减少延迟和配额消耗。当 API 负载高时，系统会自动降级到 Fast Mode：

```typescript
// Fast Mode 配置
const DEFAULT_FAST_MODE_FALLBACK_HOLD_MS = 30 * 60 * 1000 // 30分钟
const SHORT_RETRY_THRESHOLD_MS = 20 * 1000 // 20秒
const MIN_COOLDOWN_MS = 10 * 60 * 1000 // 10分钟
```

### 13.5.2 Fast Mode 降级逻辑

当 Fast Mode 遇到 429/529 错误时的处理流程：

```typescript
// 如果 429 错误是因为超出使用量限制
const overageReason = error.headers?.get(
  'anthropic-ratelimit-unified-overage-disabled-reason',
)
if (overageReason !== null && overageReason !== undefined) {
  // 永久禁用 Fast Mode，显示特定消息
  handleFastModeOverageRejection(overageReason)
  retryContext.fastMode = false
  continue
}

// 短等待：直接等待后重试
const retryAfterMs = getRetryAfterMs(error)
if (retryAfterMs !== null && retryAfterMs < SHORT_RETRY_THRESHOLD_MS) {
  await sleep(retryAfterMs, options.signal, { abortError })
  continue
}

// 长等待：进入冷却期，切换到标准速度
const cooldownMs = Math.max(
  retryAfterMs ?? DEFAULT_FAST_MODE_FALLBACK_HOLD_MS,
  MIN_COOLDOWN_MS,
)
triggerFastModeCooldown(Date.now() + cooldownMs, cooldownReason)
```

Fast Mode 的优势包括：减少 API 调用延迟；降低 token 消耗；在 API 繁忙时保持可用性。

## 13.6 避免被封号的策略

### 13.6.1 理解可能导致问题的行为

虽然 Claude Code 本身不会导致账号被封，但不当使用可能触发 API 的安全机制。以下是应该避免的行为：

**过高的请求频率**：短时间内发送大量请求可能被视为滥用；建议使用内置的重试机制，让系统自动处理限流。

**异常的使用模式**：固定间隔的机械请求可能被视为机器人；建议加入人类化的随机延迟。

**大量失败的请求**：频繁触发 429/529 错误可能表明使用方式有问题；建议优化请求批次，减少不必要的 API 调用。

**超出配额后继续尝试**：持续尝试可能触发更严格的限制；建议在收到限流通知后等待。

### 13.6.2 正确使用限流机制

Claude Code 的限流设计已经考虑到了各种场景，正确使用可以避免问题：

**依赖内置重试**：不要在代码中手动重试，让 withRetry 机制处理：

```typescript
// 错误：手动重试
for (let i = 0; i < 5; i++) {
  try {
    await apiCall()
  } catch (e) {
    await sleep(1000)
  }
}

// 正确：依赖内置重试
const result = await withRetry(
  () => getAnthropicClient(),
  async (client) => client.messages.create(params),
  { maxRetries: 10 }
)
```

**监听限流通知**：关注警告消息，提前采取措施：

```typescript
// 配额状态监听
export const statusListeners: Set<StatusChangeListener> = new Set()

export function emitStatusChange(limits: ClaudeAILimits) {
  currentLimits = limits
  statusListeners.forEach(listener => listener(limits))
}
```

**使用模型降级**：在配额紧张时使用更小的模型：

```typescript
// 配置备用模型
const fallbackModel = 'claude-3-haiku-20240307'
```

### 13.6.3 配额管理最佳实践

以下是一些实用的配额管理建议：

**定期检查配额状态**：使用 /limits 命令查看当前配额：

```bash
claude-js --help | grep limit
```

**设置使用提醒**：在配额达到 70-80% 时会收到警告。

**合理规划使用**：避免在配额周期结束时集中使用。

**使用企业级订阅**：企业用户通常有更高的配额和更宽松的限制。

### 13.6.4 错误处理最佳实践

当遇到错误时，应该遵循以下原则：

**不要立即重试**：收到 429/529 错误后，等待一段时间再重试；内置的重试机制会智能处理。

**检查错误消息**：错误消息会说明限制类型和重置时间。

**使用 session 限制**：5 小时会话限制适合短时任务，避免长时间连续使用。

**监控使用量**：定期检查 API 使用量，避免意外超额。

## 13.7 常见限流错误处理

### 13.7.1 错误代码对照表

| 错误代码 | 含义 | 处理方式 |
|---------|------|---------|
| 429 | 速率限制 | 等待后重试，系统会自动处理 |
| 529 | 服务过载 | 等待后重试，系统会自动处理 |
| 401 | 认证失败 | 检查 API 密钥配置 |
| 403 | 权限不足 | 检查账户权限设置 |
| 400 | 请求错误 | 检查请求参数是否正确 |

### 13.7.2 特定场景处理

**场景一：会话限制用尽**

```
错误消息：You've hit your session limit · resets in 2 hours
处理方式：等待或开始新会话
```

**场景二：周限制用尽**

```
错误消息：You've hit your weekly limit · resets in 5 days
处理方式：等待下周或购买额外使用量
```

**场景三：额外使用量用尽**

```
错误消息：You're out of extra usage · resets in 3 hours
处理方式：等待重置或联系管理员增加配额
```

**场景四：API 服务过载（529）**

```
错误消息：API overloaded, please retry
处理方式：系统会自动重试，等待即可
```
