# 第四章：查询引擎

## 4.1 查询引擎概述

查询引擎是 Claude Code 的大脑，负责处理用户输入、与 Anthropic API 通信、管理对话状态，以及协调工具的执行。查询引擎由三个主要组件构成：QueryEngine.ts 提供高级编排功能，管理会话级别的状态；query.ts 实现核心的对话循环，处理消息的发送和接收；API 客户端（src/services/api/claude.ts）负责与 Anthropic API 的底层通信。

这三者形成了一个分层架构，每一层都有明确的职责边界。 QueryEngine 是面向 UI 的接口，包装了底层的查询逻辑；query.ts 是核心引擎，实现了对话的循环逻辑；API 客户端是底层适配器，处理网络通信和数据格式化。这种分层设计使得各部分可以独立演进，同时保持整体的稳定性。

## 4.2 三层架构详解

### 4.2.1 第一层：QueryEngine 高级编排器

**文件**: `src/QueryEngine.ts`

QueryEngine 是高级编排器，主要负责会话级别的状态管理。它的核心职责包括：维护消息历史，跟踪对话的完整上下文；管理任务状态，处理任务的创建、更新和完成；记录权限拒绝，当用户拒绝工具调用时记录下来以便后续处理；追踪使用统计，记录 API 调用的 token 消耗；处理消息的序列化和反序列化，支持会话恢复和持久化。

QueryEngine 的关键方法包括 submitMessage()（提交用户消息并获取响应）、resume()（恢复已中断的会话）、abort()（中止当前操作）和 getState()（获取当前状态）。这些方法为 UI 层提供了清晰的接口，使得界面逻辑与核心引擎解耦。

### 4.2.2 第二层：query.ts 核心循环

**文件**: `src/query.ts`

query.ts 实现了真正的对话循环，是整个系统的心脏。它包含一个 while(true) 循环，持续处理用户输入和 API 响应，直到对话结束。

核心循环的执行流程如下：首先，构建发送给 API 的消息列表，包括系统提示、历史消息和当前输入；然后，调用 API 客户端获取响应，处理流式数据并实时 yield 给调用者；接着，检测响应中是否包含工具调用，如果有则执行工具并把结果添加到消息列表；最后，检查是否需要继续对话，如果是则进入下一轮循环。

query.ts 还负责多种错误处理和恢复机制。当收到 "prompt too long" 错误时，它会触发上下文压缩；当输出 token 不足时，它会调整输出限制并重试；当流式响应超时时，它会降级到非流式请求。

### 4.2.3 第三层：API 客户端

**文件**: `src/services/api/claude.ts`

API 客户端是 Claude Code 与 Anthropic API 之间的桥梁。它负责：消息规范化，将内部消息格式转换为 API 要求的格式；构建请求参数，包括系统提示、工具定义和各种 Beta 特性标志；处理流式响应，将原始的事件流转换为结构化消息；实现错误恢复，当流式请求失败时自动尝试非流式请求。

API 客户端使用了多种优化技术来提高性能和可靠性。缓存控制使用 prompt caching 减少重复的系统提示传递；预连接在用户开始对话前就建立网络连接；超时保护确保长时间无响应时能够正确处理。

## 4.4 流式响应处理

### 4.4.1 流式事件处理循环

**文件**: `src/services/api/claude.ts` (第 1941-2305 行)

API 客户端使用 async generator 处理流式响应。每当有新的事件到达时， generator 会 yield 一个结构化的消息片段。

```typescript
// 处理各种事件类型
for await (const event of stream) {
  switch (event.type) {
    case 'message_start':
      // 初始化消息对象，捕获首次响应时间
      break;
    case 'content_block_start':
      // 创建新的内容块容器
      break;
    case 'content_block_delta':
      // 累加内容增量（文本、工具输入、思考等）
      break;
    case 'content_block_stop':
      // 完成内容块构建
      break;
    case 'message_delta':
      // 更新消息元数据（usage、stop_reason 等）
      break;
    case 'message_stop':
      // 流结束清理
      break;
  }
}
```

这种事件驱动的处理方式使得程序可以实时响应 API 的输出，而不需要等待完整响应。

### 4.4.2 内容累加模式

流式响应的核心是累加模式，每次收到增量数据时将其添加到对应的内容块中。

```typescript
case 'input_json_delta':
  contentBlock.input += delta.partial_json; // 累加工具输入 JSON
  break;
case 'text_delta':
  contentBlock.text += delta.text; // 累加响应文本
  break;
case 'thinking_delta':
  contentBlock.thinking += delta.thinking; // 累加思考内容
  break;
```

对于工具输入，API 发送的是 JSON 片段，需要逐步累加并最终解析为完整的 JSON 对象。

### 4.4.3 超时保护机制

**文件**: `src/services/api/claude.ts` (第 1869-1930 行)

流式响应实现了双重超时保护：stall 检测和 idle 超时。

```typescript
const STALL_THRESHOLD_MS = 30000;  // 30 秒无新事件触发警告
const STREAM_IDLE_TIMEOUT_MS = 90000;  // 90 秒无任何数据触发中止

function resetStreamIdleTimer() {
  clearStreamIdleTimers();
  streamIdleWarningTimer = setTimeout(warnMs => {...}, STALL_THRESHOLD_MS);
  streamIdleTimer = setTimeout(() => {
    streamIdleAborted = true;
    releaseStreamResources();
  }, STREAM_IDLE_TIMEOUT_MS);
}
```

这种设计确保即使 API 无响应，用户也不会无限等待。

### 4.4.4 非流式回退

**文件**: `src/services/api/claude.ts` (第 2405-2708 行)

当流式响应失败时，程序会自动降级到非流式请求。

非流式回退的触发条件包括：流式空闲超时（90 秒无数据）；流式完成但没有内容块；创建流式请求时返回 404（网关不支持流式端点）；用户中止时区分 SDK 超时和用户中止。

回退机制确保了对话的可靠性，即使在网络不稳定的情况下也能完成任务。

## 4.5 工具调用执行

### 4.5.1 流式工具执行架构

**文件**: `src/query.ts` (第 560-569 行)

Claude Code 支持流式工具执行，这意味着工具可以在收到 API 响应后立即开始执行，而不需要等待完整的响应流结束。

```typescript
const useStreamingToolExecution = config.gates.streamingToolExecution;
let streamingToolExecutor = useStreamingToolExecution
  ? new StreamingToolExecutor(tools, canUseTool, toolUseContext)
  : null;
```

流式工具执行允许并行处理多个工具调用，提高了整体效率。

### 4.5.2 工具调用检测

**文件**: `src/query.ts` (第 828-848 行)

在处理 API 响应时，程序会检测是否包含工具调用。

```typescript
if (message.type === 'assistant') {
  const assistantMessage = message as AssistantMessage;
  const msgToolUseBlocks = ... .filter(content => content.type === 'tool_use');

  if (msgToolUseBlocks.length > 0) {
    toolUseBlocks.push(...msgToolUseBlocks);
    needsFollowUp = true;  // 触发下一轮循环
  }

  // 将工具加入流式执行队列
  if (streamingToolExecutor && !signal.aborted) {
    for (const toolBlock of msgToolUseBlocks) {
      streamingToolExecutor.addTool(toolBlock, assistantMessage);
    }
  }
}
```

当检测到工具调用时，程序会设置 needsFollowUp 标志，这会导致对话循环继续执行，而不是结束。

### 4.5.3 工具结果收集

**文件**: `src/query.ts` (第 850-865 行, 1386-1411 行)

工具执行完成后，结果会被收集并添加到消息列表中。

```typescript
// 收集已完成的工具结果
if (streamingToolExecutor && !signal.aborted) {
  for (const result of streamingToolExecutor.getCompletedResults()) {
    if (result.message) {
      yield result.message;
      toolResults.push(...normalizeMessagesForAPI([result.message], ...));
    }
  }
}

// 等待剩余的工具结果（异步）
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext);

for await (const update of toolUpdates) {
  if (update.message) {
    yield update.message;
    toolResults.push(...normalizeMessagesForAPI([update.message], ...));
  }
}
```

工具结果被规范化为 API 消息格式后，会被添加到消息列表中，用于下一次 API 调用。

## 4.6 对话状态管理

### 4.6.1 状态类型定义

**文件**: `src/query.ts` (第 204-217 行)

query.ts 中定义了跨迭代的对话状态。

```typescript
type State = {
  messages: Message[];  // 消息历史
  toolUseContext: ToolUseContext;  // 工具使用上下文
  autoCompactTracking: AutoCompactTrackingState | undefined;  // 自动压缩跟踪
  maxOutputTokensRecoveryCount: number;  // 输出恢复计数
  hasAttemptedReactiveCompact: boolean;  // 是否尝试过响应式压缩
  maxOutputTokensOverride: number | undefined;  // 输出限制覆盖
  pendingToolUseSummary: Promise<ToolUseSummaryMessage | null> | undefined;  // 待处理摘要
  stopHookActive: boolean | undefined;  // 停止钩子是否激活
  turnCount: number;  // 对话轮次
  transition: Continue | undefined;  // 继续原因
};
```

这些状态在对话循环中保持，确保每轮对话都能访问完整的历史信息。

### 4.6.2 消息生命周期

**文件**: `src/query.ts` (第 1717-1731 行)

每轮对话结束后，状态会更新为下一轮做准备。

```typescript
const next: State = {
  messages: [...messagesForQuery, ...assistantMessages, ...toolResults],
  toolUseContext: toolUseContextWithQueryTracking,
  autoCompactTracking: tracking,
  turnCount: nextTurnCount,
  maxOutputTokensRecoveryCount: 0,
  hasAttemptedReactiveCompact: false,
  pendingToolUseSummary: nextPendingToolUseSummary,
  maxOutputTokensOverride: undefined,
  stopHookActive,
  transition: { reason: 'next_turn' }
};
state = next;
```

这种状态转移模式确保了对话的连续性，同时为下一轮对话准备好所有必要的信息。

### 4.6.3 会话持久化

QueryEngine 负责将消息序列化到会话文件中，支持中断后恢复。

## 4.7 错误恢复机制

### 4.7.1 输出 Token 恢复

**文件**: `src/query.ts` (第 1188-1259 行)

当输出 token 不足时，程序会尝试多轮恢复。

```typescript
if (isWithheldMaxOutputTokens(lastMessage)) {
  // 第一轮：尝试增加 token 限制到 64k
  if (maxOutputTokensOverride === undefined) {
    state = { ..., maxOutputTokensOverride: ESCALATED_MAX_TOKENS };
    continue;
  }

  // 多轮恢复：递减阈值
  if (maxOutputTokensRecoveryCount < MAX_OUTPUT_TOKENS_RECOVERY_LIMIT) {
    const recoveryMessage = createUserMessage({
      content: "Output token limit hit. Resume directly...",
      isMeta: true
    });
    state = { ..., maxOutputTokensRecoveryCount: maxOutputTokensRecoveryCount + 1 };
    continue;
  }
}
```

这种递增式的恢复策略试图在保留尽可能多输出的同时完成任务。

### 4.7.2 提示过长恢复

**文件**: `src/query.ts` (第 1065-1186 行)

当提示超过模型限制时，会触发三层恢复机制。

第一层是上下文折叠 drain，这是最便宜的恢复方式，保留细粒度的上下文；第二层是响应式压缩，对对话历史进行完整摘要；第三层是错误传播，如果所有恢复都失败，则向用户报告错误。

```typescript
// 检测提示是否过长
if (reactiveCompact?.isWithheldPromptTooLong(message)) {
  withheld = true;
}

// 过滤被扣留的消息
if (!withheld) {
  yield message;
}
```

### 4.7.3 消息扣留模式

**文件**: `src/query.ts` (第 791-827 行)

程序可以使用消息扣留来处理可恢复的错误，而不是立即失败。

```typescript
let withheld = false;

// 上下文折叠
if (feature('CONTEXT_COLLAPSE')) {
  if (contextCollapse?.isWithheldPromptTooLong(message, ...)) {
    withheld = true;
  }
}

// 响应式压缩
if (reactiveCompact?.isWithheldPromptTooLong(message)) {
  withheld = true;
}

// 输出 token 扣留
if (isWithheldMaxOutputTokens(message)) {
  withheld = true;
}
```

被扣留的消息不会立即 yield 给 UI，而是尝试恢复后再处理。

## 4.8 核心技术模式

### 4.8.1 Generator-Based 异步流

所有三层都使用 async generator（async function*）实现背压敏感的流式处理。

```typescript
// API 客户端返回 generator
async function* queryModelWithStreaming(...) {
  for await (const event of stream) {
    // 处理事件，构建消息
    yield message;  // 立即流向 UI
  }
}

// query.ts 使用 yield* 委托
yield* queryLoop(...);

// QueryEngine 消费 generator
for await (const message of queryEngine.submitMessage(...)) {
  // 更新 UI
}
```

这种设计使得数据能够从 API 流畅地传递到 UI，中间经过多个处理层，而不需要等待完整的结果。

### 4.8.2 检查点仪表化

程序在关键位置插入检查点，用于性能追踪。

```typescript
queryCheckpoint('query_autocompact_start');
// ... 操作 ...
queryCheckpoint('query_autocompact_end');
```

这些检查点帮助开发者定位性能瓶颈和优化机会。

### 4.8.3 资源清理

程序使用 finally 块确保资源被正确清理。

```typescript
async function* queryLoop(...) {
  try {
    // 对话循环
    for await (const message of callModel(...)) {
      yield message;
    }
  } finally {
    // 清理资源
    cleanupStream(stream);
    releaseToolExecutor();
  }
}
```

这确保了即使发生错误，网络连接和文件句柄等资源也能被正确释放。
